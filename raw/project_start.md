## 0. 프로젝트 MD 파일 체계 (모든 프로젝트 공통)

> 새 프로젝트 시작 시 이 구조를 따른다.

```
<project-root>/
├── CLAUDE.md                  ← Claude 작업 규칙 (매 세션 자동 로드, ~100줄 이내 유지)
├── README.md                  ← GitHub용 프로젝트 소개
│
├── docs/
│   ├── ROADMAP.md             ← 세션 진입점: 현재 상태 + 다음 할 일 + 개발 연혁
│   ├── plans/                 ← 구현 계획서 (기능명-날짜.md, 확정 후 불변)
│   └── specs/                 ← 설계 명세서 (기능명-design-날짜.md, 확정 후 불변)
│
├── raw/
│   └── project_start.md       ← 범용 개발 철학·방법론 (이 파일, 다른 프로젝트에 복사)
│
└── <domain>/
    └── README.md              ← 도메인별 서브디렉토리 설명 (필요 시)

~/.claude/projects/<project-path>/memory/
├── MEMORY.md                  ← 메모리 인덱스 (200줄 이내 유지)
├── user.md                    ← 사용자 프로필 + 자동학습 규칙
├── reference-project-start.md ← raw/project_start.md 위치 포인터
└── <기타>.md                  ← 환경·참조 메모리
```

### 역할 구분 원칙

| 파일 | 독자 | 변경 빈도 |
|------|------|----------|
| `CLAUDE.md` | Claude (자동) | 아키텍처 변경 시 |
| `docs/ROADMAP.md` | 사람 + Claude | 브랜치 머지·Phase 전환 시 |
| `docs/specs/` | 설계 참조 | 확정 후 불변 |
| `docs/plans/` | 구현 참조 | 확정 후 불변 |
| `raw/project_start.md` | 다른 프로젝트 Claude | 새 패턴 발견 시 |
| `memory/MEMORY.md` | Claude (자동) | 메모리 파일 추가·삭제 시 |
| `memory/user.md` | Claude (세션 로드) | 사용자 방향성 변화 시 |

### 새 프로젝트 시작 시 체크리스트

## 1. 엔지니어링 철학

### 성능이 모든 기술 선택의 기준
기존 모델 활용, 파인튜닝, 신규 모델 개발 — 방법은 무관하다. 중요한 것은 목표 성능 달성이다.
성능 목표는 추상적("잘 동작한다")이 아닌 수치로 제시한다 (예: 정확도 95%, CER ≤ 5%).
기술 선택 추천 시 항상 성능 근거를 먼저 제시받기를 원한다.

### 파이프라인 사고
문제를 `입력 → 단계들 → 출력` 구조로 분해한다.
각 단계는 독립적으로 교체 가능해야 한다 (어댑터 패턴).
전체를 한 번에 만들지 않고, 파이프라인 구조를 먼저 확정한 후 각 단계를 채운다.

### 단계적 실체화 (Phased Delivery)
- **Phase 1:** 핵심 기능만, 제약 조건 안에서 동작 검증
- **Phase 2+:** 범위 확장, 정확도 향상, 엣지케이스 처리
- Phase 간 건너뜀 없음. 현재 Phase가 동작하기 전에 다음 Phase 구현 시작하지 않는다.

### YAGNI 실천
현재 Phase에 필요 없는 기능은 `[N단계]`로 명시하고 구현하지 않는다.
미래 요구사항을 대화 중 언급하되, 현재 구현 범위에 포함시키지 않는다.

## 2. 개발 방법론

### 전체 흐름 (게이트 방식)
```
브레인스토밍 → 설계 문서 확인 → 구현 계획 확인 → SDD 실행 → 브랜치 마무리
```
각 게이트에서 사용자 확인 후 다음 진행. 설계 없이 코드 작성 시작하지 않는다.

### 브레인스토밍 → 설계
- 질문은 한 번에 하나씩. 답 듣고 다음 질문.
- 2~3개 접근법 제안 + 권장안 + 트레이드오프 제시.
- 설계 문서 작성 → 사용자 검토 → 확인 후 구현 계획으로.

### 구현: 서브에이전트 주도 개발 (SDD)
계획 확정 → 태스크별 서브에이전트 순차 디스패치.

**실제 SDD 루프 (AIScore에서 검증된 절차):**
```
1. task-brief 스크립트로 태스크 브리프 파일 추출
2. implementer 서브에이전트 디스패치 (brief 파일 + report 파일 경로 전달)
3. review-package 스크립트로 diff 패키지 생성
4. reviewer 서브에이전트 디스패치 (brief + report + diff 세 파일)
5. Spec ✅ + Quality Approved → 레저(progress.md)에 기록
6. Critical/Important 발견 시 fix 서브에이전트 단일 디스패치 (전체 목록 한 번에)
7. 전체 태스크 완료 → 최종 whole-branch 리뷰 → fix → 브랜치 마무리
```

**Pre-flight 검토:** SDD 시작 전 계획서를 한 번 훑어 모순/버그를 배치 발견.
→ 실제 사례: Task 11 path depth 버그(`parent.parent` → `parent.parent.parent`), 파일명 오류 사전 수정.

**레저(`.git/sdd/progress.md`):** 컨텍스트 압축 후에도 재개 가능하도록 태스크별 커밋 범위 기록.

### TDD 기본
구현 전 실패 테스트 먼저 작성 → 실패 확인 → 구현 → 통과 확인 → 커밋.
단위 테스트 + 통합 테스트 + E2E 테스트 계층 구성.

### 아키텍처: 헥사고날 (포트 & 어댑터)
도메인 레이어는 순수하게 유지 (외부 라이브러리 import 금지).
교체점은 어댑터로만. 새 엔진/모델 = 새 어댑터 파일 추가, 기존 코드 수정 아님.
인터페이스(포트) 동결 후 구현 시작.

## 3. 데이터 / ML 전략

### 레이블 가용성 먼저 분석
학습 전략 결정 전에 "어떤 레이블을 확보할 수 있는가"를 먼저 따진다.
- GT 레이블 제공 가능 → 지도학습
- 레이블 확보 어려움 → 합성 데이터 자동 생성 또는 자기지도학습

### 합성 데이터 활용
레이블링 비용이 높은 경우, 프로그래밍으로 생성 가능한 합성 데이터를 우선 고려한다.
(예: MusicXML → Lilypond 렌더링으로 악보 이미지+레이블 자동 생성)

### 기준선 측정 필수
새 모델/방법을 도입할 때 기존 방법 대비 성능 측정을 먼저 한다.
"더 좋다"는 직관이 아닌 수치로 검증한다.

## 4. AI 협업 스타일

### 파트너십 지향
AI를 코드 작성 도구가 아닌 개발 파트너로 대우한다.
AI는 태스크 완료 후 다음 단계를 선제적으로 제안해야 한다.
"무엇을 도와드릴까요?" 대신 "다음은 X를 해야 합니다"가 효율적이다.

### 자율 실행 위임
계획이 확정되면 중간 체크인 없이 끝까지 실행한다.
블록되거나 모순 발견 시에만 중단하여 확인.
실행 중 "계속할까요?" 질문 불필요.

### 세션 연속성
매 세션은 현재 상태 요약으로 시작한다:
- 완료된 것 / 다음 할 일 / 보류 중인 결정

메모리는 프로젝트 상태 + 의사결정 근거를 포함해야 한다.
중요한 결정이나 패턴 발견 시 능동적으로 메모리에 저장한다.

### 능동적 메모리 기대
"기억해줘"라고 하지 않아도 중요한 방향성 변화, 새로운 패턴, 핵심 결정은 즉시 저장.
세션이 바뀌어도 같은 수준의 컨텍스트 유지.

## 5. Q&A에서 도출된 핵심 결정들 (AIScore 사례)

> 이 섹션은 추상 원칙이 아닌 실제 대화에서 나온 **구체적 결정과 그 근거**를 기록한다.
> 다른 프로젝트에서는 의사결정 패턴을 참고용으로 사용한다.

### OMR 엔진 전환 결정
- **결정:** Audiveris(규칙기반) → YOLOv8 기반 DL-OMR으로 전환
- **근거:** Audiveris 실측 65% 정확도, 목표 95% — 갭이 커서 DL 전환 불가피
- **교훈:** 기존 툴을 먼저 측정한 뒤 전환 결정. 직관이 아닌 수치.

### 학습 데이터 전략 결정
- **결정:** 음표/쉼표/기호 레이블 → Lilypond 합성 렌더링으로 자동 생성
- **근거:** "박자 레이블은 직접 제공하기 어렵다" (사용자 직접 언급) → 합성 데이터 대안 채택
- **결정:** 계이름/가사 레이블 → 찬송가 악보 GT로 지도학습 가능 (사용자 제공 가능)
- **교훈:** 레이블 가용성을 사용자가 먼저 판단한다. AI는 가용성 기반으로 전략을 제안.

### 악보 범위 결정
- **결정:** 1단계는 찬송가(hymn)만. 칸타타(cantata) PDF는 Phase 2 이후.
- **근거:** 찬송가는 단순 4성부 SATB 구조, 칸타타는 복잡도·레이아웃 다양성 높아 Phase 2 이후.
- **교훈:** "칸타타 PDF 맞죠?" → "최후에"로 즉각 범위 확정. 브레인스토밍 중 추가 요구사항은 Phase N으로 분리.

### 중간 표현 형식 결정
- **결정:** OMR 결과를 MusicXML로 직렬화, 다운스트림(SVS)에 전달
- **근거:** MusicXML은 범용 악보 교환 포맷 — music21, MuseScore, 외부 검증 도구와 호환
- **교훈:** 파이프라인 단계 간 인터페이스는 범용 표준 포맷을 사용하면 단계별 독립 검증 가능.

### SDD 방식 채택 결정
- **결정:** 태스크별 서브에이전트 자동 실행 + 리뷰 게이트 (SDD Option 1)
- **근거:** 옵션 2 (수동 실행)는 중간 체크인이 많아 속도 저하. 계획이 확정된 이후엔 자율 실행이 효율적.
- **교훈:** 계획 확정 = 실행 위임. 실행 중 "계속할까요?" 없음.

## 6. 소통 스타일

- **언어:** 한국어 기본. 코드 식별자(함수명/변수명)는 영어.
- **간결함:** 짧고 직접적인 질문·지시. 불필요한 재확인/요약 생략.
- **빠른 수렴:** 방향이 맞으면 즉시 다음 단계로. 과도한 검토 없음.
- **요구사항 추가:** 브레인스토밍 중 추가 요구사항이 나온다 → Phase N으로 분리 수용.
- **도메인 기여:** 기술 제안에 도메인 현실(데이터 가용성, 제약 등)로 검증 참여.
